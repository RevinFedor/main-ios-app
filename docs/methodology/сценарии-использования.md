# Сценарии использования

Пошаговые пути юзера через систему — переходы между состояниями (foreground ↔ background ↔ killed), точки входа (Action Button / Control Center / Shortcut / in-app UI), edge-states с прерываниями. Подсистемы участвуют по 2+: Voice + Live Activity + audio session + clipboard. Один путь не помещается ни в один отдельный `fact-*.md`.

## Сценарий: запуск записи через Action Button из killed-state

Юзер сделал force-quit app (или iOS его выгрузил после долгого фона). Хочет быстро начать запись из любого app.

### Основной путь

1. **App killed.** Process не существует, App Group `isRecording=false`.
2. **Юзер нажимает Action Button** (привязан к Shortcut «Voice Toggle»).
3. **iOS cold-launch'ит наш host app** через `LiveActivityIntent` conformance — perform() гарантированно в host process. Видим `[Coord] init — history=N` сразу за `[Intent] Shortcut toggle perform()`.
4. **perform() пишет `pendingAction=start` в App Group** + `wantsVoiceTab=true`, зовёт `handlePendingActionIfNeeded`.
5. **`AudioSessionManager.activate()`** на serial queue — `setCategory(.playAndRecord, mode: .measurement, options: [.allowBluetoothA2DP, .mixWithOthers]) + setActive(true)`. Если в фоне играла музыка через AirPods — ~500ms-1s дыра (route reconfig duplex).
6. **`Activity.request()`** для Live Activity — perform() foreground-equivalent на iOS 17+ (см. `fact-live-activity.md::Persistent .idle mode [deprecated]`).
7. **`engine.start()` + tap installed** на текущий input port (по умолчанию iPhone built-in mic).
8. **WS connect к Soniox** — async, до 1.7с.
9. **`phase → .recording`**, perform() возвращает `.result(value: "")` (пустая строка на start).
10. **Юзер видит pop-out в Dynamic Island** «Voice Record / Recording…» с timer'ом. Может говорить.

### Edge: AirPods подключены к iPhone

Тот же путь, но шаг 5 не вызывает HFP-переключения т.к. категория без `.allowBluetoothHFP`. AirPods остаются в hi-fi A2DP, mic берётся с iPhone built-in. Музыка в AirPods продолжает играть без дыры. Юзер может **явно** переключиться на AirPods-mic через picker → шаг 5 повторится с HFP-категорией → A2DP музыка остановится (см. ниже).

### Edge: нет network

Шаг 8 не завершится. Wait-for-recording loop в perform() ждёт 8с, потом возвращает `.result(value: "")`. Phase остаётся `.starting`, юзер видит `Starting…` в Live Activity. Может или ждать восстановления сети (WS auto-reconnect), или нажать Cancel в expanded Live Activity.

## Сценарий: остановка записи через Action Button с фоновым плеером

Юзер записывает, в фоне играет музыка в AirPods. Жмёт Action Button чтобы стопнуть и положить транскрипт в clipboard.

### Основной путь

1. **App backgrounded** (или foreground, или вообще killed-launched этим же intent'ом).
2. **Юзер нажимает Action Button.** Shortcut «Voice Toggle» запускается.
3. **iOS поднимает Action Button-host UI** (Shortcuts overlay) → наш scene уходит в background. Process переходит в `.suspended`-готовое состояние когда perform() завершится.
4. **perform() читает `isRecording=true` из App Group → action=stop**, пишет `pendingAction=stop`, зовёт `handlePendingActionIfNeeded`.
5. **`handlePending` (case "stop") ставит `stopOriginatedFromIntent = true`** в Coordinator (см. `fix-background-intent-crashes.md::Jetsam`).
6. **Coordinator.stop()** → `setPhase(.stopping)` (alerted Live Activity update → banner pop-out) → `dictation.stop()` шлёт **пустой TEXT frame** Soniox'у как finalize → mic stop сразу (tap removed, engine stopped — аудио после stop'а не пишется) → wait for `{finished:true}` + `ws.onclose` (нормально 1-10с, до 30с на длинной речи). Live "Finalizing · Xs tail" индикатор в toolbar пока Soniox дотягивает is_final. Hard-cap 60с страхует от frozen WS. → `fireStopped(pcm)` → `didStopWith`. См. `fact-voice-record.md::Stop & finalize lag`.
7. **`didStopWith` пишет транскрипт в stash** (`pendingClipboardText` в App Group) и пробует `UIPasteboard.general.string = text` — silent no-op в фоне.
8. **`RecordingActivityManager.end()`** показывает `.ended` Live Activity update с alertConfiguration «Готово — текст скопирован» → **зелёный banner в верху экрана / Dynamic Island pop-out** (~1.5с visible).
9. **Внутри `end()` проверяется `stopOriginatedFromIntent == true` → `setActive(false, .notifyOthersOnDeactivation)`** → AirPods переходит на A2DP-only → **~1с дыра в музыке** (визуально замаскирована banner'ом).
10. **`act.end(dismissalPolicy: .immediate)`** через 1.5с — banner исчезает.
11. **perform() читает `pendingClipboardText` из App Group** → возвращает `.result(value: transcript)`.
12. **Shortcuts.app pipe'ит value в action 2 «Copy to Clipboard»** → транскрипт в системном clipboard.
13. **Юзер открывает Notes / Messages → Paste** → текст вставляется.

### Edge: in-app stop через UI-кнопку (тот же flow, но без deactivate)

Юзер в Voice tab, жмёт большую red record button (или Cancel button). Идёт через `RecordingCoordinator.toggle()` напрямую, **минуя** `handlePending`. Флаг `stopOriginatedFromIntent` остаётся `false`. В `end()` deactivate-ветка **не** срабатывает → сессия остаётся active → нет hardware flush → нет дыры в фоновой музыке. Зелёный banner всё равно показывается (тот же `.ended` pop-out). После stop'а app foreground-active, jetsam не угрожает.

### Edge: запись была через AirPods-mic (HFP)

Шаг 9 — `reconfigureForCurrentTarget()` flip'ит категорию обратно к A2DP-only **до** `setActive(false)`. AirPods освобождают HFP-link → могут вернуться в A2DP. Фоновая музыка возобновляется.

### Edge: process suspended до того как Shortcuts pipeline закончит

Маловероятно — Shortcuts.app сама держит execution assertion. Но если случится — `pendingClipboardText` остаётся в App Group, при следующем foreground наше app'а через `flushPendingClipboard()` запишется в clipboard. Юзер просто не получит **немедленный** paste, а будет вынужден вернуться в наш app сначала.

## Сценарий: in-app start/stop с играющей музыкой

Юзер в Voice tab, музыка играет в AirPods, хочет записать заметку без прерывания музыки.

### Основной путь

1. **App foreground.** Музыка играет через A2DP.
2. **Юзер тапает большую red record button.** `RecordingCoordinator.toggle()` → `start()`.
3. **`activate()`** — `setCategory(.playAndRecord, .measurement, [.allowBluetoothA2DP, .mixWithOthers]) + setActive(true)`. Первый раз за life процесса. **Hardware flush на ~500ms — короткая дыра в музыке** (always-active pattern не помогает на первом старте, см. `fact-audio-session.md::Always-active`).
4. **`Activity.request()` + `engine.start()`**. Phase=`.recording`.
5. **Юзер говорит.** Транскрипт streaming'уется в preview через Soniox.
6. **Юзер тапает stop.** `toggle()` → `stop()` → finalize.
7. **`didStopWith` пишет clipboard напрямую** (app foreground → `UIPasteboard` работает) → `autoCopy: wrote pasteboard (foreground), N chars`.
8. **`end()` показывает зелёный `.ended` banner**, через 1.5с dismiss. `stopOriginatedFromIntent=false` → **НЕТ `setActive(false)`** → сессия остаётся active → нет hardware flush → **нет дыры**.
9. **Юзер видит транскрипт в text area Voice tab**, может его use'ать.
10. **Следующая запись на той же сессии** — шаги 4-9 без шага 3. **Никакого hardware flush, никаких дыр.** Это и есть выигрыш always-active pattern.

### Edge: AirPods отключаются во время записи

`AVAudioSession.routeChangeNotification` reason=`.oldDeviceUnavailable` → `AudioSessionManager.onActiveRouteChange` callback → `MicCaptureHub.rebuildTap()` (см. `fact-audio-session.md::Route change mid-recording` и `::MicCaptureHub`). Engine stops, tap removed, fresh tap install'ится с новым input format (iPhone built-in 48kHz), engine restarts. Gap в PCM ~100ms, оба стока (диктовка + long, если параллель) continue непрерывно. Юзер не замечает.

### Edge: прерывание сессии (звонок / Siri / будильник) или потеря сети

Юзер диктует, кнопка мика красная. Звонок/Siri/будильник на секунду деактивируют аудио-сессию (iOS останавливает движок), ИЛИ пропадает интернет и Soniox отваливается.

1. **Сеть упала:** жёлтое предупреждение «нет интернета» появляется, но **запись продолжается** — `.wav` копится независимо от WS. С возвратом сети WS reconnect'ится, live-транскрипт дотекает; если офлайн был длиннее 10 минут — часть live-текста пропущена (буфер capнут), но аудио целое и перераспознаётся из истории. Запись кончает ТОЛЬКО Стоп.
2. **Сессия прервана:** система деактивировала сессию + остановила движок. На `.ended` (или, если событие не пришло — на 2-секундном watchdog'е по `engine.isRunning`) `MicCaptureHub` делает hard-reactivate сессии + рестарт движка/тапа — кадры снова текут в ту же запись. Если юзер успел нажать Старт во время сорванной сессии и получил бы `-50` — старт сам себя лечит (hard-reactivate + retry), без перезапуска приложения.
3. **Юзер жмёт Стоп** — нормальный finalize, запись сохранена.

Механика — `fact-audio-session.md::Прерывание сессии`. Принцип «сбой зависимости ≠ стоп записи» — `methodology/переносимый-дизайн.md::Сбой зависимости не останавливает запись`.

## Сценарий: Action Button когда наш app foreground

Юзер в Voice tab, жмёт Action Button (не red button) — рефлекс с iPhone 15 Pro.

### Основной путь

1. **App foreground.**
2. **Юзер жмёт Action Button.** iOS принимает hardware-button input.
3. **iOS показывает Shortcuts host UI overlay** → наш scene уходит в background activation state.
4. Дальше **идентично** «Сценарий: остановка через Action Button» начиная с шага 4. Флаг `stopOriginatedFromIntent` = true (потому что `handlePending` всегда устанавливает).
5. **deactivate в end() срабатывает** → если играла фоновая музыка, ~1с дыра.
6. **Юзер ожидает что Action Button сработает как in-app stop button** — это **отдельное несоответствие UX** которое мы решили принять. Hardware Action Button **физически** уводит фокус на Shortcuts host, наш scene **физически** становится background — невозможно различить «реально в фоне» от «foreground-app + Action Button» на уровне ОС. См. `fix-background-intent-crashes.md::Graveyard` про неудачные попытки detect'ить foreground через applicationState.

### Урок

Если юзер находится в нашем app и **видит** record-button — он должен использовать **его**, не Action Button. Action Button — для ситуаций «я в Notes/Messages, хочу диктовать». Это разное interaction-modality, не дублирование.

## Сценарий: параллельная запись — long в фоне + диктовка поверх

Юзер запускает long-запись (фоновый аудио-дамп), затем поверх неё — обычную диктовку. Обе пишут с одного мика, сохраняются отдельно. Это основной flow параллельной записи (`fact-voice-record.md::Параллельная запись`, `fact-audio-session.md::MicCaptureHub`).

### Основной путь

1. **App foreground, Voice tab.** Оба слота idle.
2. **Юзер тапает `recordingtape` (long-кнопка).** `Coordinator.toggleLong()` → `startLong()`. `longPhase=.starting`. `MicCaptureHub.attach(longSink)` — **первый** сток → активирует сессию + стартует движок. `longSink` (transcribe:false) сразу репортит connected → `longPhase=.recording`. Live Activity: `longStart()` создаёт Activity, NC-карточка «Ожидает команды…», Island пустой.
3. **Long-панель на главном экране** показывает иконку + (по long-press) таймер от `longStartedAt`.
4. **Юзер тапает центральную mic-кнопку** (или угол Control Center, или Action Button — все три = диктовка). `start()`. `dictationPhase=.starting`. `MicCaptureHub.attach(dictationSink)` — **второй** сток, движок УЖЕ крутится → мгновенный attach, без второго `setActive`. Live Activity: `dictationStart()` **adopt'ит** ту же Activity, штампует диктовочный chrome ПОВЕРХ (long-поля сохранены), Island показывает диктовку (приоритет).
5. **Обе записи идут с одного мика.** Tap хаба ресэмплит раз, fan-out'ит в оба стока — каждый копит свой `.wav`. Soniox стримит только диктовочный сток.
6. **Юзер стопит диктовку** (центральная кнопка). `stop()` → finalize Soniox → `didStopWith(session===dictationSession)`: сохраняет транскрипт, `dictationSink` **detach** от хаба. Движок **продолжает крутиться** (long-сток ещё attach'ен). `dictationEnd()`: long ещё активен → Activity НЕ dismiss, диктовочный слот → `.idle`, NC откатывается на «Ожидает команды…».
7. **Long продолжает писать** как ни в чём не бывало. Юзер может стопнуть диктовку и запустить её снова сколько угодно раз поверх одного long.
8. **Юзер стопит long** (■ Stop в панели). `stopLong()` → `didStopWith(session===longSession)`: сохраняет `.wav` под `запись_NNN`, `longSink` **detach** — это **последний** сток → хаб стопит движок. `longEnd()`: диктовка idle → Activity dismiss'ится.

### Edge: обратный порядок (диктовка первой, long поверх)

Симметрично. Диктовка стартует Activity с диктовочным chrome; `longStart()` видит существующую Activity → лишь выставляет `longActive=true` в её ContentState (Island остаётся диктовочным, приоритет). Когда диктовка стопнется первой — Activity переключится на long-chrome.

### Edge: jetsam-деактивация при параллели

Критическая тонкость. Intent-stop диктовки (Action Button) обычно делает `setActive(false)` против jetsam (`fix-background-intent-crashes.md::Jetsam`). **Но если long ещё пишет — деактивировать сессию НЕЛЬЗЯ**, это вырвет route из-под long-стока. Поэтому `dictationEnd` гейтит `setActive(false)` дополнительным условием «long-слот idle». При живом long intent-stop диктовки остаётся jetsam-незащищённым — но этот путь возможен только с foreground-приложением (long нельзя стартовать из killed-state, у него нет intent-входа), где jetsam не угрожает.

### Edge: смена мика во время параллели

Карточка выбора встроенного мика (Низ/Перед/Зад) дизейблится по `isCapturing` (любой слот) — менять hardware-мик нельзя пока хоть одна запись держит роут. А вот выбор iPhone↔AirPods (через `reapplyLiveTarget`) сменит роут для ОБЕИХ записей разом (мик общий) + триггерит rebuild tap в хабе на новом sample rate — оба стока продолжают непрерывно.

## Сценарий: long-запись пережила убийство приложения → восстановление

Юзер запускает long-запись, сворачивает/гасит телефон, потом **убивает приложение** свайпом в переключателе (или его выгружает jetsam под память, или краш) — БЕЗ нажатия Stop. Ожидает: запись не потеряна, на следующем открытии он понимает, что произошло. Механика — `fact-voice-record.md::Long пишется на диск` и `::Восстановление осиротевшей long-записи`.

### Основной путь

1. **App foreground.** Юзер тапает `recordingtape` → `startLong()`. `.wav` открыт на диске СРАЗУ (placeholder-заголовок), в App Group лёг маркер `{path, title, start}`. С этого момента каждый кадр стримится на диск.
2. **Юзер сворачивает приложение** (Home / другое приложение / гаснет экран). Запись идёт в фоне (`UIBackgroundModes=audio` + активная сессия удерживают процесс), кадры продолжают литься в файл.
3. **Юзер убивает приложение свайпом вверх** в переключателе задач. Процесс умирает мгновенно, кода «меня закрывают» НЕ исполняется. Но на диске уже лежит весь звук, кроме последнего кадра; маркер в App Group **переживает** смерть процесса (это persistent storage, не память).
4. **Юзер заново открывает приложение (cold-launch).** `RecordingCoordinator.init` → `recoverOrphanedLong` видит выживший маркер ⇒ был kill без Stop. Чинит заголовок `.wav` из размера файла, `append`'ит запись в Long с **исходным** временем старта (из маркера), публикует `recoveredLong`.
5. **На Voice-экране под «Open History» — оранжевая плашка** «`запись_007` остановлена в 15:52 · 4:32 / приложение закрыто без остановки записи». Время = mtime файла (последний кадр до kill). Запись уже в Long, играется.
6. **Юзер тапает плашку (или открывает Историю любым путём).** Открывается История, плашка гаснет. Запись на месте.

### Edge: чистый Stop (плашки НЕ должно быть)

Если long остановлен нормально (■ Stop в панели), `didStopWith` патчит заголовок через `writer.finish()` и **чистит маркер**. Следующий cold-launch не видит маркера → `recoverOrphanedLong` — no-op, плашки нет. Это разделитель «убит» vs «остановлен»: единственный признак — выжил ли маркер.

### Edge: восстановление не удалось (файл пуст / пропал)

Маркер есть, но файл отсутствует или меньше 44 байт (kill в первую секунду, до первого кадра). `recoverOrphanedLong` чистит маркер **первым делом** (до проверки файла) — поэтому сбой не зацикливается на каждом запуске, — затем тихо выходит без плашки. Лучше молча ничего, чем битая запись в истории.

## Сценарий: Chat из Voice-экрана — надиктовал, выбрал промпт, ушёл в чат

Юзер надиктовал заметку и хочет не копипастить её, а сразу скормить агенту на Mac с
выбранным промптом — и оказаться в живом чате с результатом. Проходит через Voice + пикер +
сеть до Mac + вкладку AI Chat + SSE. Подсистемы: `fact-voice-chat-tab.md` (iOS-обёртка),
`voice-record/fact-mobile-bridge.md` (сервер).

### Основной путь

1. **Voice tab, запись остановлена, есть транскрипт.** Под ним ряд Copy/Notes/Share и
   кнопка **Chat**.
2. **Юзер тапает Chat.** Транскрипт **снимается в снимок** в момент тапа (переживёт, если
   фоном начнётся новая запись).
3. **Снизу выезжает bottom-sheet пикер промптов** (не fullscreen). Шит сам грузит
   `/api/prompts` со спиннером. В header видны model, thinking level и быстрый `Switch`
   между двумя mobile presets; это именно Voice-поверхность отправки диктовки в новый чат.
   Первая строка — «Отправить без промпта», дальше prompt-группы раскрываются по тапу, а
   листовой промпт выбирается.
4. **Юзер выбирает промпт.** `POST /api/chat/send {text: транскрипт, promptId?, model,
   thinkingLevel, bypass}` на Mac. На корне приложения (над табами) — капсула «Создаю чат…»;
   двойной тап заблокирован.
5. **Сервер резолвит промпт в чип + текст, создаёт чат, запускает `runAgent`, возвращает
   `chatId`.**
6. **Приложение переключает таб на AI Chat** на этот `chatId` (через `router.openChat`).
   Нижнее меню (AI Chat · Voice · Remote · Habits) **остаётся видимым** — это вкладка, не
   модалка.
7. **Нативная вкладка AI Chat загружает conversation через `VoiceChatStore`**:
   `GET /api/chats/:id` даёт историю и pending-confirm, общий SSE догоняет live-события.
   Лента идёт как prompt-чип → сообщение → thinking/tool-карточки (агент реально лезет
   в MCP/bash на Mac) → ответ.
8. **Юзер может продолжать чат** прямо в нативной вкладке или свободно уйти на другой таб
   и вернуться — вкладка тёплая, разговор на месте.

### Edge: диктовка завершилась, пока открыт AI Chat

Юзер уже находится на странице конкретного AI Chat (не в drawer и не на All chats), диктует
через Voice toggle / Action Button / Control Center и завершает запись, не переводя фокус в
composer. После finalize Voice публикует готовый transcript, а открытый chat detail
принимает его как continuation текущего разговора: текст вставляется в конец composer draft,
не затирая уже набранное. Фокус в `TextEditor` не требуется — source of truth здесь страница,
на которой находится пользователь. Если в этот момент открыта история, transcript не
вставляется скрыто: история — поверхность выбора разговора, не активный consumer текста.

### Edge: prompt выбран внутри уже открытого AI Chat

Если юзер находится в AI Chat detail и открывает prompt picker из composer, выбор prompt'а
не запускает новый чат и не отправляет turn. Prompt становится фиолетовым chip'ом текущего
draft; Send остаётся активным даже при пустом input, потому что сам chip уже является
полезным payload. Это намеренно отличается от Voice/history Chat-кнопки, где выбор prompt'а
сразу отправляет снимок диктовки в новый разговор.

### Edge: Mac недоступен на шаге 4

POST падает `OfflineError` (туннель жив, app выключен → 503; или нет сети). Капсула гаснет,
показывается ошибка создания **на корне** (переживает переключение таба — не на скрытом
Voice). Юзер остаётся на Voice с транскриптом, повторяет позже.

### Edge: создание прошло, но переход «не случился» (был баг)

Гонка `@Published` (`pendingChatId` против `selected`) могла потерять переход — таб
оставался на старом разговоре. Лечится UUID-токеном запроса + реакцией на оба сигнала, см.
`fact-voice-chat-tab.md::Handoff openChat`.

### Edge: «Отправить без промпта»

Шаг 4 без `promptId` — только транскрипт. Сервер требует непустой text|prompt; пустой
транскрипт + без промпта отсекается.

### Edge: By-pass меняется во время стрима

By-pass в iOS toolbar — не снимок на момент отправки. Если агент дошёл до tool-confirm,
переключение вызывает `/api/chat/bypass`, сервер шлёт SSE `bypass`, а `confirm()` перед
каждым tool-call перечитывает живой per-chat state. OFF оставляет confirm-карточки на
экране, ON применяет следующий tool-call и закрывает pending; после Allow/Deny или bypass
true клиент tombstone'ит локальные `callId`, чтобы reconnect не воскресил старые карточки.

### Edge: Stop после thinking/tool

Если Stop нажат до первого assistant artifact, chat возвращает input в composer и не
оставляет пустой ответ. Если уже появился хотя бы thinking-блок или tool-карточка,
остановка сохраняет частичный assistant message как обычный ответ со stopped-состоянием:
пользователь видит, что агент уже успел сделать, а не получает «стёртый» ход.

## Сценарий: Terminal Stop сразу после Send — draft возвращается из TUI

Юзер управляет custom-terminal из нативного AI Chat и быстро отменяет только что отправленный
prompt. Путь проходит через SwiftUI Terminal composer, custom-terminal REST/SSE, Claude/Codex
PTY runtime и фактический TUI composer. Подсистемы: `fact-voice-chat-tab.md` (Terminal mode),
`custom-terminal/docs/knowledge/fact-mobile-claude-pty.md` и `fact-mobile-web.md`.

### Основной путь

1. **AI Chat → Terminal mode → выбран проект и Claude/Codex tab.** В composer набран prompt
   или выбран terminal prompt/GT-chip.
2. **Юзер жмёт Send.** iOS очищает composer как pending turn, но store отмечает tab как
   recoverable; отправленный текст не становится источником будущего восстановления.
3. **custom-terminal пишет prompt в PTY/CLI.** Loader/thinking на телефоне ждёт runtime event,
   а не таймер.
4. **Юзер быстро жмёт Stop.** Для Claude это `/interrupt` → одиночный Esc в PTY; для Codex
   runtime ждёт rollout `turn_aborted`.
5. **iOS не возвращает prompt немедленно.** Сначала должен прийти deterministic reason:
   `interrupt`, `turn_aborted` или `catch-up-turn_aborted`.
6. **После reason iOS читает `/api/sdk-tabs/:id/draft` несколько раз короткими probes.**
   Если CLI реально вернул текст в TUI composer, этот текст вставляется обратно в mobile
   composer. Пользователь видит тот же draft, который теперь есть у desktop runtime.

### Edge: draft не вернулся

Если `/draft` пустой после probes, mobile показывает нейтральное notice «отмена подтверждена;
terminal input пустой» и оставляет composer пустым. Нельзя подставлять last-sent prompt из
памяти: это нарисует текст, которого нет в terminal composer и который не будет виден самому
CLI при следующем действии.

## Сценарий: вопрос агента при выключенном By-pass — разрешить/отклонить с телефона

Юзер хочет контролировать мутации (правки файлов, bash) на Mac, не находясь за ним. Проходит
через composer → сервер (парковка confirm) → SSE → закреплённую карточку → ответ → продолжение
turn'а. Подсистемы: `fact-voice-chat-tab.md` (карточки, tombstones),
`voice-record/fact-mobile-bridge.md::By-pass и confirm` (серверная парковка).

### Основной путь

1. **AI Chat, By-pass OFF** (toolbar pill). Юзер отправляет задачу, требующую мутации
   («создай файл на рабочем столе»).
2. **Агент на Mac доходит до `write_file`/`bash`** → сервер паркует вопрос
   (`pendingConfirms`) и шлёт SSE `confirm`. Turn стоит — ждёт ответа.
3. **На телефоне появляется жёлтая карточка** в закреплённой зоне НАД композером (не в
   ленте — видимость не зависит от скролла): действие, путь/команда, мини-дифф,
   «Разрешить»/«Отклонить».
4. **Юзер отвечает.** Allow → инструмент исполняется, turn продолжается; Deny → агент
   получает «rejected by user» и завершает ход без мутации. Карточка снимается, callId
   уходит в tombstones.

### Edge: юзер ушёл с вкладки, пока вопрос висит

Переключение на Voice/Habits замораживает SSE-сокет (iOS, без ошибки — зомби-пайп). При
возврате `start()` видит stale-стрим по тишине heartbeat'ов → реконнект + re-hydrate;
карточка восстанавливается из `pendingConfirms` в `GET /api/chats/:id`. Пока юзер
отсутствовал, turn честно ждал (таймаут вопроса 10 мин → deny).

### Edge: устаревший GET против свежего SSE (был баг «завис на минуту»)

GET диалога, сериализованный ДО парковки вопроса, мог приехать ПОСЛЕ SSE-карточки и
затереть её; восстановленная поллингом карточка оказывалась ниже вьюпорта. Лечение —
закреплённая зона + merge `union(local, server) − tombstones`, см.
`voice-record/docs/methodology/переносимый-дизайн-mobile.md::Вопрос агента — закреплённая зона`.

### Edge: Stop при висящем вопросе

`/api/chat/stop` отклоняет все запаркованные вопросы чата (иначе агент ждал бы ответа,
который никогда не придёт после abort'а) и обычным образом откатывает/сохраняет ход.
