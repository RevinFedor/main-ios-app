# Принципы UX и дизайна

Переносимые принципы организации пользовательского взаимодействия — выработаны в этом проекте, но применимы в любом приложении с похожими симптомами.

## Прежде чем строить workaround — проверить актуальную документацию платформы

Раньше в этом файле жил «Persistent system-state объект» принцип — про обход «Activity.request() forbidden from background» через постоянно живую `.idle` Activity. **Принцип оказался построен на неверной предпосылке.** Apple ActivityKit docs прямо говорит: *«you can't call Activity.request() while your app is in the background, unless you adopt App Intents and start the Live Activity using a LiveActivityIntent»*. Constraint снят с iOS 17, наш intent уже conform'ит `LiveActivityIntent`, workaround не нужен.

Урок не «вот хороший паттерн» а «**проверь свежие docs прежде чем городить workaround**». Особенно для платформ с быстрой эволюцией (iOS major release раз в год). Persistent-mode добавлял visible system-state индикатор (Dynamic Island compact icon постоянно) — серьёзный UX-cost — ради решения проблемы которая уже не существовала. Когда constraint вернётся (BLE-кнопки, CLLocationManager wakes, background processing) — тогда и думать о workaround. До тех пор `Activity.request()` напрямую из `LiveActivityIntent.perform()` работает без побочек.

В проекте: workaround снят целиком, см. `knowledge/fact-live-activity.md::Persistent .idle mode [deprecated]`.

## Hardware constraints отображать в UI рядом с действием

Когда юзер выбирает опцию в UI, и выбор влечёт **hardware-level** последствие которое юзер не может предположить из названия опции — об этом последствии надо сказать **в момент выбора**, не после.

В проекте: picker микрофона в Voice tab показывает два пункта — «Микрофон iPhone» и «AirPods Pro». Под пунктом AirPods надпись «*музыка остановится*» серой подстрокой. Это потому что Bluetooth-протокол **физически** не позволяет одновременно держать A2DP-стрим (стерео музыка) и HFP-стрим (микрофон) с одного устройства. Без подписи юзер тапает «AirPods» ожидая что просто переключится input, и удивляется когда фоновая музыка обрывается. Подпись делает trade-off видимым в момент решения.

Юзер прямо спросил: *«почему AirPods отключают музыку?»* — это сигнал что constraint существует но **невидим** в UI. Принцип шире: любое interaction которое триггерит system-level reconfiguration (BT route, camera focus mode, screen brightness override, focus mode disable) должно либо иметь inline-warning, либо confirmation alert. См. `knowledge/fact-audio-session.md::Category-by-target`.

## Выбирать дают то, что относится к задаче приложения

Прежде чем добавлять селектор устройства/режима — проверь, влияет ли этот выбор на то, **ради чего приложение существует**. Класс приложения определяет какая сторона железа релевантна: для рекордера это **вход** (микрофон), для плеера — **выход** (динамик). Селектор не-релевантной стороны выглядит логично («у звука же есть и вход и выход»), но юзеру он бесполезен и путает.

В проекте сделали и тут же удалили выбор динамика (speaker/receiver) на экране записи: при записи звук **никуда не выводится**, приложение только пишет с микрофона — выбирать выход не из чего и незачем. Юзер прямо: *«выбор динамика вообще нахуй не нужен... при записи звук никуда не выводится»*. Заменили на выбор встроенного микрофона (Низ/Перед/Зад) — это **вход**, он реально меняет результат записи. Тест перед добавлением селектора: «изменит ли этот выбор артефакт, который производит главная функция приложения?». Нет → селектор не нужен, сколько бы технических опций ни отдавала платформа. См. `knowledge/fact-audio-session.md::Output-выбор динамика`.

## Sticky-дефолт — не клетка

Закреплённое-по-умолчанию значение задаёт стартовую точку, но **не должно блокировать** ручной выбор другого варианта. Юзер ожидает что «закреп» = «открывается на этом по умолчанию», а не «больше ничего нельзя выбрать». Если закреп физически дизейблит альтернативы — это читается как баг («не могу выбрать X»).

В проекте: лок «Закрепить iPhone-мик» — sticky через запуски, но строка AirPods в меню осталась **выбираемой**; тап по ней снимает закреп и переключает на AirPods. Раньше строка была `.disabled(locked)` — и юзер сообщил ровно симптом-клетку: *«не могу выбрать AirPods если закреплён микрофон iPhone... закреп открывается по умолчанию, но если выбираю AirPods — закреп пропадает»*. Правило: выбор любой альтернативы автоматически снимает sticky-флаг (override переписывает default), а не требует сперва вручную «расстегнуть» закреп. Реализация — `knowledge/fact-voice-record.md::Mic-picker` (закреп = DEFAULT, не клетка).

## Background-trigger = system suspend через секунды, ресурсы надо отдать

iOS убивает app которое **суспендится** удерживая system resources (audio session, mic, camera, location significant-change subscription). Watchdog — отдельный от memory jetsam: SIGKILL через 2-10 секунд после suspension. Если background-launched action (Action Button intent, Control Center toggle, Shortcut from outside app) триггерит финализацию работы — в **финале** action'а ресурс должен быть отдан, иначе app не успеет до watchdog.

В проекте: `RecordingActivityManager.end()` при `stopOriginatedFromIntent=true` зовёт `AVAudioSession.setActive(false, .notifyOthersOnDeactivation)` перед dismiss Activity. Иначе — `0x8badf00d` через ~3-7с, новый cold-launch видно по `[Coord] init — history=N+1` в логах. Цена deactivate — hardware route reconfig ~1с (BT A2DP↔HFP renegotiation на AirPods, см. Apple QA1631), это маскируется visible `.ended` banner'ом так что юзер видит как часть success animation, не как unexplained pause.

Тонкость: distinction «foreground action vs background action» не делается через `applicationState` (`LiveActivityIntent.perform()` поднимает scene в `.active` даже без UI). Только через **точку входа**: in-app UI-button → `toggle()` напрямую → no deactivate; Action Button / Shortcut → `handlePendingActionIfNeeded` → ставит флаг → deactivate. См. `knowledge/fix-background-intent-crashes.md::Jetsam`.

Принцип шире чем audio: любая background-triggered финализация (закрыть сетевое соединение, отписаться от location, выпустить camera-permission) должна **всегда** идти в финале action'а, не оптимизироваться через «может в следующий раз использую — пусть подождёт». Apple watchdog не понимает «may reuse».

## Stealth-mode для системных индикаторов записи

Иконки и тексты UI, связанные с активной записью микрофона (mic-glyph, «Recording», «Voice Record»), должны появляться **только когда запись реально идёт** — то есть когда AVAudioSession (или эквивалент) активна и iOS показывает свой системный privacy-indicator. В состояниях «приложение готово, ничего не пишется» эти элементы заменяются на нейтральные.

Юзер: *«я не хочу, чтобы это отображалось на моём экране, особенно чтобы другие пользователи не подумали, что я их записываю на фоне. Поэтому значок микрофона в верхнем менюшке замени на какой-нибудь другой, который не связан с аудио»*. Это privacy-concern не от paranoid'а, а от рационального отношения к тому что bystander видит на чужом телефоне.

В проекте принцип сейчас не воплощён в runtime, потому что persistent idle Activity (которая раньше требовала stealth UI) была убрана как ненужный workaround (см. секцию выше «Прежде чем строить workaround»). Но stealth-маппинг остался в `VoiceRecordingLiveActivity.swift` view-функциях для `.idle` phase — title=«Habit Tracker», phaseLine=«Приложение активно», все mic-glyphs (compact/minimal/Lock Screen/NC) заменены на `sparkles` (читается как «AI / smart action», семантически нейтральная). Если когда-нибудь persistent-режим вернётся (BLE-кнопки, location wakes — см. fact-live-activity.md::Persistent .idle mode [deprecated]) — stealth-маппинг уже готов.

Принцип шире чем recording app: любой UI-element который **сейчас не делает** того что подразумевает его icon — лжёт пользователю и bystander'у. Microphone icon без активного mic, location-pin без активной геолокации, camera-icon без активной камеры — все это enabled by stealth-mode паттерну.

Тот же принцип в обратную сторону — **выбор метафоры иконки по тому, что элемент означает, а не по домену**. Юзер про navbar-индикатор аудиосессии: *«измени его иконку, чтобы она не была связана со звуком, а такая, типа, иконка статуса»* — индикатор показывал `speaker.wave`/`speaker.slash`, но его смысл не «звук играет», а «сессия прогрета / выключена» (готовность, не воспроизведение). Динамик-глиф наводил на ложную мысль про playback. Заменили на `checkmark.circle`/`circle.dotted` — нейтральный статус готовности. Урок: иконка кодирует **семантику состояния**, а не техническую подсистему за ним; «тут про аудио → нарисуем динамик» — ловушка, динамик означает вывод звука, а не «готовность к записи».

## Серийные действия не запирать за анимацией завершения

Когда действие предполагается выполнять **сериями** (переставить десять строк подряд, отметить пачку чекбоксов, быстро пролистать), нельзя заставлять юзера ждать окончания анимации завершения одного действия прежде чем он сможет начать следующее. Юзер: *«меня бесит то, что мне нужно ждать, пока произойдёт анимация, кнопка отпустится, и только после этого я смогу зажать следующий элемент… я не могу их быстро перетаскивать… я работаю очень быстро»*. Анимация-фидбек хороша как подтверждение, но если она блокирует следующий ввод — на серии она превращается в налог, умноженный на количество действий.

Тонкость: иногда задержка зашита в системный компонент, а не в твой код. Нативный `List.onMove` на iOS держит drag на `UICollectionView`-распознавателе с неотключаемым ~0.5s lock после дропа — и единственный способ убрать налог был отказаться от нативного механизма в пользу своего жеста, который не запирается. То есть прежде чем принять «так работает компонент» — проверь, не противоречит ли его встроенная анимация серийному паттерну использования; если да, кастомная реализация без lock'а оправдана. См. `knowledge/fact-habit-tracker.md::Перестановка`.

## Gestures-first: убираем edit mode целиком

Классический режим «Edit / Карандаш» — paradigm перенесённая с десктопа, где не было long-press и drag. На современном iOS жесты долгого нажатия и перетаскивания юзер уже знает с домашнего экрана (иконки, виджеты). Дублировать эти жесты явной кнопкой «Edit» — двойная работа: визуальный шум + ментальный шаг «сначала войду в режим, потом сделаю».

Решение — *long-press + drag всегда доступны*, отдельной кнопки/режима нет. Список становится «прозрачным» в терминологии Apple HIG: пользователь видит контент, действует напрямую, нет «прослойки» режима. В проекте это применено к списку привычек — long-press 0.3s открывает шторку редактирования, удержание + drag меняет порядок и переносит между группами. См. `knowledge/fact-habit-tracker.md`.

## Один канал на действие — не дублировать кнопку и контекст-меню

Если действие уже доступно в long-press контекст-меню, не держи его ещё и inline-кнопкой на карточке — это один и тот же выбор, разложенный в два места, визуальный шум без новой возможности. Юзер про карточку истории сказал прямо: *«при клике открывается менюшка Copy / Return Scribble / Delete, но при этом ещё есть повторяющиеся кнопки — кнопки на самой карточке убери»*. Контекст-меню стало каналом для этой тройки, на карточке остались только действия которые в меню НЕ дублируются (Play, объединение). Тест на дубль: «это действие уже достижимо другим жестом на этом же элементе?» — да → оставь один канал, не два.

Тот же принцип шире — про **избыточные обучающие affordance**: когда стиль элемента уже сигналит его природу, не вешай дополнительный значок-подсказку. Юзер про дату-chip: *«убери карандашик рядом с датой, я и так помню что она редактируется»* — chip-оформление (пилюля с фоном) само читается как «тут можно тапнуть», иконка-карандаш поверх уже-узнаваемого паттерна повторяет сообщение, которое контекст донёс. Affordance оправдан пока паттерн новый; как только он закрепился (все editable-поля выглядят одинаково) — значок становится шумом. В проекте оба случая — `knowledge/fact-voice-record.md::История`.

## Confirm в nav-bar + dirty-state, шит на часть экрана

Форма редактирования (sheet) следует трём платформенным конвенциям, которые юзер сформулировал по ходу. Confirm-действие живёт **в nav-bar справа**, отмена/закрытие — слева, не нижней floating-кнопкой: *«кнопку сохранить внизу убери, перенеси в верхний угол справа, слева будет кнопка закрыть»*. Confirm меняет вид по наличию изменений — обычная пока форма нетронута, выделенная (bold+enabled) как только данные правлены: *«если я какие-то данные изменил, то кнопка становится другого вида»*. Это стандартный «у тебя есть несохранённые правки» affordance — считается через сравнение текущих полей с исходными, а не через факт фокуса в поле. И шит открывается **на часть высоты**, не на весь экран: *«где-то на 60%, не на всю высоту, как это обычно в Apple работает»* (`presentationDetents` с partial + large).

Отдельная ловушка из той же сессии: не оптимизируй закрытие шита через отключение dismiss-анимации. Попытка убрать ~0.35s slide-down через `disablesAnimations` дала **визуальное зависание** при живой клавиатуре/баре (*«какое-то зависание, странно работает по состоянию»*). Эталон — системный Settings-шит с обычным `dismiss()`; если он «открывается корректно», копируй его поведение, а не изобретай ускорение. См. `knowledge/fact-habit-tracker.md::Edit/Add sheets`.

## Нативный компонент vs кастом: цена ухода с платформы

Когда нативный компонент почти подходит, но не даёт желаемой формы — соблазн заменить его на кастом велик, а цена скрыта. Кастом-замена тащит за собой баги платформы, которые нативный компонент глушит изнутри, и ручную возню с тем, что нативный делал бесплатно. В проекте это всплыло дважды на свайп-кнопках истории. Сначала нативный `.swipeActions` не умел круглые вертикальные кнопки (только горизонтальные прямоугольники во всю высоту) — сделали кастомный свайп-reveal на `ScrollView + LazyVStack`. Он заработал, но: (1) уход с `List` разбудил iOS-18/26 баг, где кастомный `DragGesture` на строке ломает вертикальный скролл (нативный `List` его не имеет); (2) пришлось руками мерить ширину панели, считать порог full-swipe, тюнить haptic. Юзер в итоге: *«сделай как раньше, нативные… но не круги»* — откатили на `.swipeActions`. Тот же сюжет был с порогом full-swipe (нативные 75% против желаемых 60% — нельзя настроить, не уходя в кастом): тогда юзер тоже выбрал «оставить нативное».

Правило: оставайся на нативном пока возможно, и считай реальную цену кастома до того как строить — не «сколько кода», а «какие гарантии платформы я теряю» (скролл-арбитраж, accessibility, корректные анимации, поведение на новых OS). Прототип кастома допустим как способ узнать цену, но дефолт — откат на нативное, если визуальная выгода не перевешивает потерю гарантий. Кастомный путь не выбрасывается из головы — фиксируется как «рабочий, но требует возни» (см. `knowledge/fix-ios-stability.md::SwiftUI List`), чтобы при реальной необходимости вернуться к нему осознанно.

Та же логика относится к platform chrome, который выглядит как «просто стекло». Toolbar,
search и navigation buttons надо помещать в настоящие системные контейнеры, а не рисовать
похожий `HStack` поверх контента. В iOS 26 Liquid Glass — это не только blur/material, а
поведение контейнера: группировка items, scroll-edge читаемость, lifecycle поиска, clear/cancel
кнопки, клавиатура, treatment символов. Custom glass оправдан только когда control
действительно живёт вне системного chrome; тогда всё равно брать платформенный style
(`.buttonStyle(.glass)` / аналог), а не собирать круг/капсулу вручную. В проекте это
зафиксировано на AI Chat search/settings — см. `knowledge/fact-voice-chat-tab.md::iOS 26 chrome`.

Когда системный компонент не подходит из-за продуктовой композиции, кастом не становится
лицензией на ручную имитацию платформы. Например, bottom search dock в AI Chat не может быть
`.searchable`, потому что поле должно быть постоянно раскрыто под `New Chat`, а сама кнопка
создания скрывается во время поиска. Но поверхность всё равно должна брать iOS 26
`glassEffect`/`GlassEffectContainer`, а не opacity-fill, и отдельно владеть touch-зоной:
если плавающий control визуально поверх списка, но не перехватывает hit-test всей своей
областью, тап по его `x`-кнопке проходит в строку под ним. Сначала выбирается правильная
семантика компонента, затем правильный платформенный слой, и только потом рисуется форма.

Но бывает обратный случай: откат на нативное **невозможен**, потому что само встроенное поведение компонента и есть проблема (нативный `List.onMove` с неотключаемым post-drop lock'ом против серийного reorder). Тогда нельзя и наслаивать кастомный жест **на уровне фреймворка-обёртки** — на iOS 18+ внутренний pan ScrollView'а приватный, и SwiftUI-жест через любой модификатор (`.gesture`/`.simultaneousGesture`/`.sequenced`) не сосуществует с ним детерминированно: три попытки подряд убивали скролл. Правильный кастом в этом случае спускается на **слой распознавателей самой платформы** — `UIGestureRecognizerRepresentable` + свой `UIGestureRecognizer`, чей делегат `shouldRecognizeSimultaneouslyWith` возвращает `true` для системного pan'а. Там арбитраж scroll-vs-жест ведёт UIKit (как в нативном списке), а кастомным остаётся только то, ради чего уходили. Юзер сам нащупал направление после двух провалов обёртки: *«проще сделать iOS-овский… который внутри iOS встроен в компонент»* — то есть не отказ от платформы, а спуск на её более низкий, контролируемый слой. Тест выбора слоя: «приватный ли системный жест, с которым я конфликтую?». Да → распознаватель-слой (UIKit), не жест-слой (SwiftUI). В проекте оба режима списка привычек так и сделаны — см. `knowledge/fact-habit-tracker.md::Обычный режим` и `::Перестановка`.

Для контейнерных переходов правило ещё жёстче: если жест двигает целые поверхности, а не одну
строку, он принадлежит root-контейнеру. Drawer, где одна панель выезжает поверх другой и
конфликтует с вложенными `ScrollView`, search/text input, системным back-swipe и tab bar, не
лечится `DragGesture` на background-view. Его надо проектировать как interactive container
transition: один root recognizer, явный intent-lock, разные правила для closed/open state и
заранее описанная политика «кто выигрывает» у вертикального scroll. Этот же root-владелец
жеста обязан отменять дочерние tap-actions на время pan'а: горизонтальный drag по поверхности
не должен после release срабатывать как tap по строке, которая оказалась под пальцем. В
проекте это зафиксировано для AI Chat history drawer после research: regular width —
`NavigationSplitView`, compact iPhone sliver-drawer — `UIGestureRecognizerRepresentable` +
кастомный pan recognizer; для нативного Terminal back-swipe тот же принцип реализован через
локальный suppress selection, см. `knowledge/fact-voice-chat-tab.md::Terminal mode внутри AI Chat`.

## Нативный рендер поверх чужого протокола

Если продукт уже имеет web-поверхность, но native app нужен как постоянный мобильный клиент,
не надо переносить WebView как UI-модель. Правильная граница: backend/protocol/lifecycle
остаётся у владельца продукта, а native app рисует платформенный интерфейс поверх этого
контракта. Иначе телефон наследует web-композицию, чужие safe-area/scroll проблемы и JS-
рендер, хотя пользователь ожидает iOS toolbar, drawer, sheet, кнопки и привычную инерцию
скролла. При этом native не должен переизобретать бизнес-движок: REST/SSE, session ids,
busy/done, prompt queue и history parsing приходят из источника, который уже отвечает за
детерминированность.

Тест границы: если состояние является фактом удалённого процесса или desktop-store
(идёт ли turn, какой session id, какие project icons/status marker заданы, какая очередь
armed) — оно приходит по protocol. Если это расположение кнопок, форма sheet, место floating
action или жест drawer — это native UI-решение, подчинённое платформе. В проекте так устроен
Terminal mode внутри AI Chat: custom-terminal остаётся владельцем REST/SSE и JSONL/rollout
логики, а Habit Tracker рисует SwiftUI-клиент вместо WKWebView; см.
`knowledge/fact-voice-chat-tab.md::Terminal mode внутри AI Chat`.

## Постоянный affordance, не появляющийся при интеракции

Если визуальный признак (скругление, рамка, фон) появляется только в момент взаимодействия — на каждом тапе/long-press элемент **скачет** из одной формы в другую, и это читается как баг, а не как фидбэк. Юзер про карточки истории: *«везде при взаимодействии появляются круглые рамки, которых нет в нормальном состоянии… сделай чтобы оно всегда было, а то какое-то некрасивое дёрганье»*. Конкретно: iOS при long-press поднимает строку контекст-меню со скруглёнными углами, а в покое углы были прямые — отсюда скачок. Решение — дать форму **постоянно** (скруглённый фон + `clipShape` всегда) и согласовать с ней превью-форму контекст-меню, чтобы переход был незаметен. Принцип шире свайпов: любой признак, который интеракция «проявляет», лучше показывать всегда в спокойном виде — фидбэком делать его усиление (тень, подсветку), а не само появление формы. В проекте — `knowledge/fact-voice-record.md::История`.

## Направленное слияние: инициатор вторичен, цель держит идентичность

Когда две сущности сливаются в одну и пользователь сам инициирует слияние, направление должно следовать его ментальной модели «куда вливаю», а не симметричному «усредни обе». Инициатор (та, на чьей кнопке/стрелке нажали) — **вторичен**: его содержимое дописывается в конец цели, а **цель** сохраняет идентичность — id, имя, дату, позицию в списке. Юзер про объединение заметок: *«приоритет отдаётся той заметке, в которую идёт мердж; та, на которой я нажал стрелочку — вторична»*. До этого слияние было симметричным (склейка по времени, усреднённая дата, новый id) — и оно ощущалось неправильно, потому что результат прыгал на новую позицию и терял заголовок цели. Правило переносимо на любой merge-UI (заметки, задачи, теги, ветки): спроси «что из двух юзер считает главным» — это та, куда вливают, она и держит идентичность; вторая растворяется в её хвосте. В проекте — `knowledge/fact-voice-record.md::История` (`mergeDirectional`).

## Структурный шов: богатый в UI, плоский в экспорте

Когда несколько сущностей склеиваются в одну, место склейки несёт информацию — «здесь кончилась одна часть и началась другая» — и у этой информации две формы представления: **внутри приложения** шов показывается явным разделителем, чтобы пользователь видел структуру и понимал, как собрана запись; **при копировании/экспорте** тот же шов уплощается до нейтрального разделителя (одна пустая строка), потому что наружу уходит чистый текст без приложенческой разметки. Юзер про объединённые заметки: *«в месте склеивания отображался разделитель, чтобы я понимал как это работало… при копировании это будет обычный пробел, одна пустая строка, а в приложении пометить вертикальной чёрточкой»*.

Ключ — хранить шов как **невидимый инвариантный маркер** внутри данных (control-char, который не встречается в пользовательском тексте), а не как готовую разметку: тогда одно и то же место рендерится богато в одном контексте и плоско в другом, и ни один потребитель текста не должен знать про другой. Правило переносимо на любой контент, собранный из кусков (склеенные заметки, объединённые задачи, merged-ветки диалога, конкатенация записей): граница между кусками — это данные, а не оформление; оформление выбирается на месте показа. Тонкость — каждый потребитель «плоской» формы (буфер, заголовок, счётчик длины, свёрнутое превью) обязан флэтить маркер, иначе control-char протекает в UI. В проекте — `knowledge/fact-voice-record.md::Шов склейки` (`mergeMarker`).

## Сегодняшний день — фоновый якорь, не жирный шрифт

В плотной сетке (день недели + точки прогресса × N привычек, всё на 2×2 виджете) жирный шрифт заголовка дня недостаточно заметен — глаз не сразу находит «где я нахожусь во времени». Решение — *фоновый круг `Color.blue.opacity(0.25)` под буквой текущего дня*. Это создаёт визуальный якорь, читается мгновенно даже на 100×100 px виджета.

Принцип шире: когда в плотной сетке нужно выделить «сейчас» / «здесь» / «выбранное» — фон работает лучше чем typography weight. Bold/colored text размывается в окружающем тексте на маленьком экране, fill-shape под элементом — нет.

## Визуальный вес — по ценности контента, не по роли поля

Размер и яркость текста распределяются по тому, **что важнее прочитать**, а не по формальной роли поля. «Заголовок» по привычке хочется сделать крупным и ярким просто потому что он называется заголовком — но если по смыслу основной контент это другое поле, иерархию надо инвертировать: контентное поле делается доминирующим, а метку-идентификатор отличают не размером, а более тихим оформлением (приглушённый цвет, меньший кегль, позиция). Юзер про форму привычки: *«заголовок выделяется больше, чем описание, за счёт шрифта — должно быть наоборот… пометить его по-другому, не размером»* — то есть короткое имя привычки тут лейбл, а описание это тело, и тело должно перетягивать взгляд. Демотированный заголовок (мельче + серый) начинает читаться как подпись к заметному описанию, и пара перестаёт конкурировать за внимание. Тест: спроси «что пользователь реально читает на этом экране», и отдай ему вес — даже если это поле формально «вторичное» по названию. В проекте — `knowledge/fact-habit-tracker.md::Edit/Add sheets` (заголовок 14pt secondary / описание 17pt).

## contentMarginsDisabled — отступы по содержимому, не по системе

iOS даёт виджетам стандартные `16pt` отступы со всех сторон. На 2×2 (≈155×155 px) это съедает ~32 px по обеим осям → остаётся ~120×120 для контента. В контентно-плотных виджетах (сетка дней × привычек) это сразу теряет половину строк.

Решение — `contentMarginsDisabled()` + явные внутренние padding'и `4-8pt`. Возвращает контроль над геометрией: пишешь под реальный размер виджета. Trade-off — нужно вручную поддерживать минимальный читаемый отступ от краёв (Apple Liquid Glass tinting расплывается без padding'а). В проекте: 14pt для medium, 8pt для small. См. `knowledge/fact-habit-widget.md`.

## Возвращаемое место — это вкладка, не модалка поверх всего

Когда экран — **место, куда юзер возвращается** (чат, лента, история переписки), он должен
быть вкладкой/страницей в общей навигации, а не fullscreen-модалкой поверх всего. Модалка
правильна для одноразового сфокусированного акта (выбор, подтверждение, форма — открыл,
сделал, закрыл); место, где живут и накапливаются, ей быть не может, потому что модалка
прячет нижнюю навигацию и оставляет один выход «закрыть», тогда как из места нужно свободно
уходить и приходить, не теряя его. Юзер на чат-как-модалку отреагировал висцерально-
навигационно: *«как мне вернуться обратно… нарушена навигация… внизу должна быть менюшка,
как любое другое приложение… а оно открывается на весь экран, это ошибка»* — жалоба не про
внешний вид, а про то, что место без постоянной навигации перестаёт быть местом. Тест:
«сюда возвращаются или это одноразовый акт?» — возвращаются → вкладка с сохранённой нижней
навигацией; одноразовый акт → модалка/шит. В проекте чат сперва открыли `fullScreenCover`
(меню исчезало, возврата нет) → переделали в полноценный таб AI Chat; кнопка Chat на Voice-
экране теперь **переключает таб**, а не накрывает экран. См. `knowledge/fact-voice-chat-tab.md`.

## Keyboard composer как overlay, не reflow списка

В мобильном чате клавиатура и input-композер воспринимаются как временный слой ввода поверх
истории, а не как причина перестроить всю историю. Если пользователь читает старое место,
тапает input и открывает клавиатуру, список сообщений не должен прыгать вверх только потому,
что снизу появилась клавиатура. Двигаться должен сам composer вместе с клавиатурой; история
остаётся в той же позиции, пока пользователь явно не нажал scroll-to-bottom или не начал
скроллить. Пользователь сформулировал это прямо: клавиатура с input «как бы поверх чата»,
а история «вверх не должна сдвигаться».

Это правило не означает произвольную ручную анимацию. Keyboard transition должен брать
геометрию и timing из платформенного источника и синхронизироваться с ним; если framework
даёт неполный контракт (как iOS 26 private keyboard curve), диагностика и fallback живут
на уровне platform integration, а не в визуальном тюнинге padding'ов. Если app-owned action
сам закрывает клавиатуру, fallback transition можно запускать до позднего system callback,
но длительность калибруется по device logs, а не по ощущению или чужому snippet'у. В проекте:
см. `knowledge/fact-voice-chat-tab.md::Composer keyboard` и
`knowledge/fix-ios-stability.md::iOS 26 keyboard notifications`.

## Связанные настройки переключаются одним действием

Если пользователь регулярно меняет несколько параметров как одну мысль, не заставляй его
каждый раз проходить два-три независимых select'а. Такая связка становится preset'ом:
настраивается в settings, имеет понятное имя/слот, а в рабочем месте переключается одним
tap'ом. Юзер сформулировал это про модели чата прямо: ему нужны две пары model + thinking
level, чтобы «в один клик быстро переключаться, а не тыкать отдельно на каждый Select».
Принцип переносим на любые часто повторяемые пары настроек: quality + speed, mode + tone,
engine + budget. Если параметры выбираются вместе больше одного раза, это уже режим, а не
набор отдельных контролов.

Второй край того же правила — не размазывать один и тот же mode-control по дочерним
поверхностям. Контрол живёт там, где он меняет ближайшее действие. В Voice prompt picker
model/think/Switch нужны, потому что пользователь прямо отправляет диктовку в новый чат; в
prompt picker внутри уже открытого AI Chat эти controls лишние, потому что composer уже
показывает текущий режим. См. `knowledge/fact-voice-chat-tab.md::Mobile model presets`.

Та же библиотека выбора может иметь разную commit-семантику в зависимости от поверхности.
Если picker открыт как продолжение уже существующего draft/composer, выбор должен подготовить
чип и оставить финальное решение кнопке Send; если picker открыт как часть одноразового action
из готового артефакта, выбор может сразу выполнить действие. Не делать «один компонент =
одинаковый commit везде»: одинаковая база данных prompt'ов не означает одинаковое намерение
пользователя. В проекте — `knowledge/fact-voice-chat-tab.md::Пикер промптов self-loading`.

## Плавающее действие не становится строкой списка

Если действие визуально живёт поверх списка — floating button, FAB, quick compose, `New Chat`
— оно не должно менять порядок и финальную позицию строк списка. Content-row остаётся там,
где его диктуют данные: последняя строка идёт сразу после последнего элемента, а floating
action кладётся поверх неё с учётом safe area и reachability. Когда под overlay добавляют
искусственный spacer «чтобы кнопка не перекрывала контент», пользователь видит уже не overlay,
а странно приподнятый список: последний row зависает над кнопкой и кажется отдельным pinned
блоком. Отступ нужен только как минимальная touch-доступность, не как попытка превратить
плавающее действие в часть layout-flow.

В проекте это проявилось на AI Chat history: `All chats` — обычная строка после recent-чатов,
а `New Chat` — белая floating-кнопка справа снизу. Если высота `New Chat` участвует в
content-flow, `All chats` оказывается над ней и нарушает ментальную модель «последний
элемент истории → потом поверх страницы действие создания». См. `knowledge/fact-voice-chat-tab.md::История`.

## Сбой удалённой поверхности — нативный recovery-экран, контрол восстановления на основной поверхности

Когда экран рендерит контент, которого может не оказаться (удалённый сайт во встроенном браузере, картинка по сети, видеопоток), у компонента есть «пустое» состояние, которое по умолчанию выглядит как поломка — белая или чёрная заглушка без объяснения. Это состояние надо перехватить и заменить **нативным экраном-объяснением**: что недоступно, почему (если знаешь), и кнопка повторить. Юзер про вкладку Remote: *«чтобы не отображался чёрный экран… если сайт недоступен — что класс недоступен, кнопка перезагрузки, вот это вот всё»*. Пустой кадр читается как «приложение сломалось»; тот же кадр с иконкой, адресом и кнопкой — как «связь пропала, вот что делать».

Второе требование того же эпизода — **где живёт кнопка восстановления**. Не в настройках: *«это такие кнопки, которые не в настройках, а в Намбаре стоят… на главной странице должно быть»*. Контрол, чинящий **сейчас видимый** сбой (перезагрузить, переподключиться), обязан быть на той же поверхности, где сбой виден — на основном экране или в его навбаре, в один тап. Закопать reload в настройки — значит заставить юзера, упёршегося в недоступную страницу, уйти с неё, вспомнить где переключатель, вернуться. Настройки — для конфигурации (сменить адрес, почистить данные), а не для реакции на текущее состояние. Тест: «чтобы исправить то, что я вижу прямо сейчас, сколько поверхностей надо сменить?» — ответ должен быть ноль. В проекте — `knowledge/fact-remote-tab.md::Offline / failure-обработка` и `::Кнопки в навбаре`.

## Терминальная точка операции всегда даёт видимый вердикт — и успех, и провал

Если у операции есть видимое подтверждение успеха (зелёный баннер «сохранено»), то у её **провала** обязан быть такой же видимый сигнал — иначе пользователь, привыкший к зелёной плашке, читает её отсутствие как «наверное, всё ок» и теряет данные молча. Успех и неуспех — две ветки одной терминальной точки, и асимметрия «успех показываем, провал глотаем» хуже, чем вообще никаких уведомлений: она создаёт ложную уверенность. Юзер: *«после успешной записи у нас зелёное уведомление сверху — надо такое же если запись не получилась или ошибка. может инета не было или ещё что… крч если запись не успешна — выводить всегда ошибку, чтобы пользователь понимал»*.

Практически это значит: каждый терминальный исход операции проходит через один узел, который выносит вердикт, и у этого узла перечислены **все** ветки — успех, и каждый класс провала отдельной понятной формулировкой («нет сети», «текст не распознан», «не удалось сохранить»), а не общий «что-то пошло не так». Молчание — не валидный исход для точки, на которой пользователь ждёт результат. Отдельная ветка — намеренная отмена самим пользователем: её вердиктом глушим (он сам это сделал, сообщать не о чем), и это единственное легитимное молчание. Правило переносимо на любую операцию с ожидаемым результатом (сохранение, отправка, синхронизация, экспорт): спроси «если это сейчас провалится — пользователь узнает, или экран просто останется прежним?». В проекте — `knowledge/fact-voice-record.md::Verdict-уведомление`.

## Долговечная запись + сверка-при-возврате бьёт ненадёжный сигнал-в-момент-события

Когда долгий процесс может быть прерван внезапно (приложение убито, вкладка закрыта, сеть упала), есть соблазн повесить спасение на **момент прерывания** — «при закрытии покажи уведомление / сохрани». Это ненадёжно по своей природе: среда не гарантирует исполнение кода в этот момент. iOS при свайп-убийстве в переключателе не даёт процессу отработать «меня закрывают»; браузерный `beforeunload` душится и пропускается; kill -9 не исполняет ничего. Юзер просил ровно сигнал-в-момент: *«можем сделать уведомление если приложение исполня закрываю… запись остановлена»* — и правильный ответ не «сделаем», а «в момент закрытия ненадёжно, поэтому спасаем иначе».

Развязка из двух частей. **Первая — данные пишутся долговечно ПО ХОДУ, а не на финале**: результат материализуется на диск инкрементально (каждый кадр записи сразу в файл), так что внезапная смерть теряет в худшем случае последний квант, а не всё. **Вторая — сверка при следующем возврате, а не сигнал в момент потери**: оставляем persistent-маркер «операция начата», и на следующем старте, если маркер выжил, понимаем «нас убили посреди» и достраиваем результат + показываем неблокирующую плашку «вот что было восстановлено». Сигнал приходит когда пользователь снова здесь и может на него отреагировать, а не в момент, когда его, возможно, уже нет. Плашка-информер гасится, как только пользователь дошёл до объекта, который она анонсирует (открыл Историю) — она отслужила. Тот же каркас переносим на черновики-автосейв (редактор: пишем по ходу, на следующем открытии «восстановить несохранённое?»), на фоновые задачи, на любую длительную операцию: durable-по-ходу + reconcile-on-return, не notify-on-close. В проекте — `knowledge/fact-voice-record.md::Long пишется на диск` и `methodology/сценарии-использования.md::long-запись пережила убийство`.

## Временно-инертный контрол не затемняем

Когда кнопка должна быть **временно нереагирующей** (контекст сейчас не позволяет действие), но при этом остаётся легитимной частью интерфейса — её нельзя дизейблить штатным `.disabled()`. На платформе этот модификатор не только блокирует тап, он ещё и **затемняет** контрол, а это читается как «сломано / недоступно навсегда», а не «сейчас не тот момент». Юзер про центральную mic-кнопку во время длинной записи: *«она должна быть по функционалу неактивной… но по дизайну не должна быть деактивной, затемнённой»* — то есть функция и вид это две независимые оси, и `.disabled()` склеивает их насильно. Развязка: инертность держать `guard` внутри обработчика тапа (ранний `return`), а вид оставить полноцветным. Кнопка выглядит живой, но проглатывает тап — ровно то, что нужно когда действие будет доступно «вот-вот» (здесь — будущие параллельные записи), и преждевременно сигналить «недоступно» неправильно. В проекте — `knowledge/fact-voice-record.md::Long полностью самодостаточен` (bigRecordButton).

## Жест под вес действия: раскрытие/деструктив — удержание, лёгкое — тап

Тяжесть жеста должна соответствовать весу и обратимости действия. Действие, которое разворачивает скрытый слой или может что-то испортить, вешается на **долгое нажатие** — оно требует намерения и не срабатывает от случайного касания. Лёгкое/обратимое (свернуть обратно, отменить выбор) — на обычный **тап**. Юзер про иконку длинной записи: *«раскрытие по клику не должно работать, только по зажатию… там встроенная фишка с iPhone есть»* — он апеллирует к системному паттерну (press-and-hold для контекст-меню, превью, перегруппировки иконок на home screen), который пользователь уже знает телом. Полезен **асимметричный** жест: раскрытие — hold, сворачивание — tap; так свёрнутый элемент не открывается от мимолётного касания, но закрывается мгновенно. Тонкость реализации — для надёжной тактильной отдачи на таком жесте нужен прогретый генератор (свежесозданный глохнет на первом импульсе), см. `knowledge/fix-ios-stability.md::Haptic на свежем генераторе`. В проекте — `knowledge/fact-voice-record.md::Панель long` (long-press разворачивает, tap сворачивает).

Если long-press меняет selection-state, а не просто открывает меню, пользователь всё равно
должен получить платформенный «я удерживаю» feedback: лёгкое сжатие/подсветку во время press
и haptic в момент фиксации выбора. Без этого задержка выглядит как лаг, а не как намеренный
жест. В проекте — GT file picker в `knowledge/fact-voice-chat-tab.md::GT Editor файлы как composer chips`.

## Состояние под-режима не протекает в общие индикаторы

Когда у приложения появляется параллельный режим (вторая разновидность записи, фоновая задача, альтернативный поток), его состояние должно жить **внутри собственного компонента**, а не подсвечивать каждый общий индикатор, который технически реагирует на «что-то происходит». Юзер про длинную запись: *«вся long-запись именно внутри этой иконки, её состояние… элементы максимально независимы»* — то есть навбар-заголовок, бейдж на таб-баре, цвет соседних кнопок не должны менять вид из-за этого режима, даже если у них есть условие, которое формально истинно («идёт запись → покажи REC»). Иначе один режим расползается визуальным шумом по всему экрану, и пользователь не может отделить «что относится к этому действию» от «что относится к приложению вообще». Практически это значит гейтить каждый общий индикатор по **конкретному** виду активности (`activeKind == .dictation`), а не по общему флагу (`isRecording`), и собирать всё управление режимом в одном месте. Это та же мысль о разделении ответственности, что в коде, но применённая к UI-поверхностям: у каждого режима свой «дом», состояние не утекает наружу. В проекте — `knowledge/fact-voice-record.md::Long полностью самодостаточен`.

## Сбой зависимости не останавливает пользовательскую запись — останавливает только явный Стоп

Длительный захват, которым владеет пользователь (запись звука, видео, ввод формы), не должен прекращаться оттого, что отвалилась **вспомогательная** зависимость — сеть, внешний сервис распознавания, аплоад. Эти подсистемы могут падать и подниматься сколько угодно, но единственное событие, которое завершает захват — это **явный Стоп от пользователя**. Юзер сформулировал это сам, когда диктовка молча перестала писаться после жёлтого предупреждения «нет интернета»: *«не должно иметь значения, вылазит у нас ошибка интернета или нет… она может вылазить, но это не означает остановку записи; остановка должна быть только после того, как я нажму кнопку Стоп»*. Предупреждение об отвалившейся зависимости — это информер, а не команда на завершение; смешивать «сообщить о деградации» и «прекратить работу» нельзя.

Практически это значит развязать два потока: **источник истины записи** (буфер/файл на диске) копится непрерывно и не зависит ни от какого сетевого состояния, а сетевой слой — отдельная необязательная дорожка, чей обрыв логируется и показывается, но не трогает захват. Тонкость с обратной стороны: раз зависимость необязательна, её буфер-для-повтора нельзя растить без предела в офлайне — иначе «не падаем от потери сети» оборачивается падением от исчерпания памяти; такой буфер капается по времени с отбросом старого, а полный артефакт остаётся на диске. И симметрично — если зависимость с возвратом сети может прислать частичный результат, разрыв надо переустановить (reconnect), а не считать концом записи. Тот же каркас переносим на любой захват с дополнительным сервисом поверх (запись с live-транскриптом, форма с автосохранением в облако, аплоад с превью): локальная истина живёт сама, удалённый слой — отсоединяемая надстройка, и только пользователь решает, когда всё закончилось. В проекте — `knowledge/fact-audio-session.md::Прерывание сессии` и `knowledge/fact-voice-record.md::Long пишется на диск` (источник истины — файл, сеть/Soniox — отдельная дорожка).

## Documentation pattern: docs_search вместо обязательного чтения шлюзов

Антипаттерн в `CLAUDE.md`: блок `<instructions>` который заставляет нейронку прочитать заранее заданный набор файлов в начале каждой сессии (`architecture.md`, `main-feature.md`, etc.). Это создаёт два сорта проблем. Во-первых, файлы быстро устаревают как «карта» — нейронка добавляет новое в knowledge/, но никто не обновляет шлюз. Во-вторых, шлюз — это каталог ссылок, который правила документации прямо запрещают: «CLAUDE.md не должен содержать каталог ссылок на knowledge-файлы — для этого `docs_search`».

Правильная активация — *семантический роутер `docs_search` сам подтягивает 2-4 релевантных файла по симптому запроса*. CLAUDE.md содержит только guard rails (anti-patterns, стек, философские принципы уровня всего проекта), точка. Юзер сформулировал прямо: *«что за хуйня, у нас по правилам документации main»*. То есть `main-*` файлы — артефакт ранней структуры до того как `docs_search` стал основным каналом обнаружения.

В этом проекте: миграция выполнена в текущей сессии. Удалены `architecture.md`, `main-design.md`, `main-feature.md`, `features/`, `ai-research/`. Содержимое разошлось по `knowledge/fact-*` и `methodology/переносимый-дизайн.md` по принципу симптома. CLAUDE.md переписан под Gold Standard v7.0 — `<instructions>` теперь вызывает `docs_search`, не предписывает чтение конкретных файлов.
