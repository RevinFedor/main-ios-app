# Принципы UX и дизайна

Переносимые принципы организации пользовательского взаимодействия — выработаны в этом проекте, но применимы в любом приложении с похожими симптомами.

## Прежде чем строить workaround — проверить актуальную документацию платформы

Раньше в этом файле жил «Persistent system-state объект» принцип — про обход «Activity.request() forbidden from background» через постоянно живую `.idle` Activity. **Принцип оказался построен на неверной предпосылке.** Apple ActivityKit docs прямо говорит: *«you can't call Activity.request() while your app is in the background, unless you adopt App Intents and start the Live Activity using a LiveActivityIntent»*. Constraint снят с iOS 17, наш intent уже conform'ит `LiveActivityIntent`, workaround не нужен.

Урок не «вот хороший паттерн» а «**проверь свежие docs прежде чем городить workaround**». Особенно для платформ с быстрой эволюцией (iOS major release раз в год). Persistent-mode добавлял visible system-state индикатор (Dynamic Island compact icon постоянно) — серьёзный UX-cost — ради решения проблемы которая уже не существовала. Когда constraint вернётся (BLE-кнопки, CLLocationManager wakes, background processing) — тогда и думать о workaround. До тех пор `Activity.request()` напрямую из `LiveActivityIntent.perform()` работает без побочек.

В проекте: workaround снят целиком, см. `knowledge/fact-live-activity.md::Persistent .idle mode [deprecated]`.

## Hardware constraints отображать в UI рядом с действием

Когда юзер выбирает опцию в UI, и выбор влечёт **hardware-level** последствие которое юзер не может предположить из названия опции — об этом последствии надо сказать **в момент выбора**, не после.

В проекте: picker микрофона в Voice tab показывает два пункта — «Микрофон iPhone» и «AirPods Pro». Под пунктом AirPods надпись «*музыка остановится*» серой подстрокой. Это потому что Bluetooth-протокол **физически** не позволяет одновременно держать A2DP-стрим (стерео музыка) и HFP-стрим (микрофон) с одного устройства. Без подписи юзер тапает «AirPods» ожидая что просто переключится input, и удивляется когда фоновая музыка обрывается. Подпись делает trade-off видимым в момент решения.

Юзер прямо спросил: *«почему AirPods отключают музыку?»* — это сигнал что constraint существует но **невидим** в UI. Принцип шире: любое interaction которое триггерит system-level reconfiguration (BT route, camera focus mode, screen brightness override, focus mode disable) должно либо иметь inline-warning, либо confirmation alert. См. `knowledge/fact-audio-session.md::Category-by-target`.

## Background-trigger = system suspend через секунды, ресурсы надо отдать

iOS убивает app которое **суспендится** удерживая system resources (audio session, mic, camera, location significant-change subscription). Watchdog — отдельный от memory jetsam: SIGKILL через 2-10 секунд после suspension. Если background-launched action (Action Button intent, Control Center toggle, Shortcut from outside app) триггерит финализацию работы — в **финале** action'а ресурс должен быть отдан, иначе app не успеет до watchdog.

В проекте: `RecordingActivityManager.end()` при `stopOriginatedFromIntent=true` зовёт `AVAudioSession.setActive(false, .notifyOthersOnDeactivation)` перед dismiss Activity. Иначе — `0x8badf00d` через ~3-7с, новый cold-launch видно по `[Coord] init — history=N+1` в логах. Цена deactivate — hardware route reconfig ~1с (BT A2DP↔HFP renegotiation на AirPods, см. Apple QA1631), это маскируется visible `.ended` banner'ом так что юзер видит как часть success animation, не как unexplained pause.

Тонкость: distinction «foreground action vs background action» не делается через `applicationState` (`LiveActivityIntent.perform()` поднимает scene в `.active` даже без UI). Только через **точку входа**: in-app UI-button → `toggle()` напрямую → no deactivate; Action Button / Shortcut → `handlePendingActionIfNeeded` → ставит флаг → deactivate. См. `knowledge/fix-background-intent-crashes.md::Jetsam`.

Принцип шире чем audio: любая background-triggered финализация (закрыть сетевое соединение, отписаться от location, выпустить camera-permission) должна **всегда** идти в финале action'а, не оптимизироваться через «может в следующий раз использую — пусть подождёт». Apple watchdog не понимает «may reuse».

## Stealth-mode для системных индикаторов записи

Иконки и тексты UI, связанные с активной записью микрофона (mic-glyph, «Recording», «Voice Record»), должны появляться **только когда запись реально идёт** — то есть когда AVAudioSession (или эквивалент) активна и iOS показывает свой системный privacy-indicator. В состояниях «приложение готово, ничего не пишется» эти элементы заменяются на нейтральные.

Юзер: *«я не хочу, чтобы это отображалось на моём экране, особенно чтобы другие пользователи не подумали, что я их записываю на фоне. Поэтому значок микрофона в верхнем менюшке замени на какой-нибудь другой, который не связан с аудио»*. Это privacy-concern не от paranoid'а, а от рационального отношения к тому что bystander видит на чужом телефоне.

В проекте принцип сейчас не воплощён в runtime, потому что persistent idle Activity (которая раньше требовала stealth UI) была убрана как ненужный workaround (см. секцию выше «Прежде чем строить workaround»). Но stealth-маппинг остался в `VoiceRecordingLiveActivity.swift` view-функциях для `.idle` phase — title=«Habit Tracker», phaseLine=«Приложение активно», все mic-glyphs (compact/minimal/Lock Screen/NC) заменены на `sparkles` (читается как «AI / smart action», семантически нейтральная). Если когда-нибудь persistent-режим вернётся (BLE-кнопки, location wakes — см. fact-live-activity.md::Persistent .idle mode [deprecated]) — stealth-маппинг уже готов.

Принцип шире чем recording app: любой UI-element который **сейчас не делает** того что подразумевает его icon — лжёт пользователю и bystander'у. Microphone icon без активного mic, location-pin без активной геолокации, camera-icon без активной камеры — все это enabled by stealth-mode паттерну.

## Gestures-first: убираем edit mode целиком

Классический режим «Edit / Карандаш» — paradigm перенесённая с десктопа, где не было long-press и drag. На современном iOS жесты долгого нажатия и перетаскивания юзер уже знает с домашнего экрана (иконки, виджеты). Дублировать эти жесты явной кнопкой «Edit» — двойная работа: визуальный шум + ментальный шаг «сначала войду в режим, потом сделаю».

Решение — *long-press + drag всегда доступны*, отдельной кнопки/режима нет. Список становится «прозрачным» в терминологии Apple HIG: пользователь видит контент, действует напрямую, нет «прослойки» режима. В проекте это применено к списку привычек — long-press 0.3s открывает шторку редактирования, удержание + drag меняет порядок и переносит между группами. См. `knowledge/fact-habit-tracker.md`.

## Один канал на действие — не дублировать кнопку и контекст-меню

Если действие уже доступно в long-press контекст-меню, не держи его ещё и inline-кнопкой на карточке — это один и тот же выбор, разложенный в два места, визуальный шум без новой возможности. Юзер про карточку истории сказал прямо: *«при клике открывается менюшка Copy / Return Scribble / Delete, но при этом ещё есть повторяющиеся кнопки — кнопки на самой карточке убери»*. Контекст-меню стало каналом для этой тройки, на карточке остались только действия которые в меню НЕ дублируются (Play, объединение). Тест на дубль: «это действие уже достижимо другим жестом на этом же элементе?» — да → оставь один канал, не два.

Тот же принцип шире — про **избыточные обучающие affordance**: когда стиль элемента уже сигналит его природу, не вешай дополнительный значок-подсказку. Юзер про дату-chip: *«убери карандашик рядом с датой, я и так помню что она редактируется»* — chip-оформление (пилюля с фоном) само читается как «тут можно тапнуть», иконка-карандаш поверх уже-узнаваемого паттерна повторяет сообщение, которое контекст донёс. Affordance оправдан пока паттерн новый; как только он закрепился (все editable-поля выглядят одинаково) — значок становится шумом. В проекте оба случая — `knowledge/fact-voice-record.md::История`.

## Сегодняшний день — фоновый якорь, не жирный шрифт

В плотной сетке (день недели + точки прогресса × N привычек, всё на 2×2 виджете) жирный шрифт заголовка дня недостаточно заметен — глаз не сразу находит «где я нахожусь во времени». Решение — *фоновый круг `Color.blue.opacity(0.25)` под буквой текущего дня*. Это создаёт визуальный якорь, читается мгновенно даже на 100×100 px виджета.

Принцип шире: когда в плотной сетке нужно выделить «сейчас» / «здесь» / «выбранное» — фон работает лучше чем typography weight. Bold/colored text размывается в окружающем тексте на маленьком экране, fill-shape под элементом — нет.

## contentMarginsDisabled — отступы по содержимому, не по системе

iOS даёт виджетам стандартные `16pt` отступы со всех сторон. На 2×2 (≈155×155 px) это съедает ~32 px по обеим осям → остаётся ~120×120 для контента. В контентно-плотных виджетах (сетка дней × привычек) это сразу теряет половину строк.

Решение — `contentMarginsDisabled()` + явные внутренние padding'и `4-8pt`. Возвращает контроль над геометрией: пишешь под реальный размер виджета. Trade-off — нужно вручную поддерживать минимальный читаемый отступ от краёв (Apple Liquid Glass tinting расплывается без padding'а). В проекте: 14pt для medium, 8pt для small. См. `knowledge/fact-habit-widget.md`.

## Documentation pattern: docs_search вместо обязательного чтения шлюзов

Антипаттерн в `CLAUDE.md`: блок `<instructions>` который заставляет нейронку прочитать заранее заданный набор файлов в начале каждой сессии (`architecture.md`, `main-feature.md`, etc.). Это создаёт два сорта проблем. Во-первых, файлы быстро устаревают как «карта» — нейронка добавляет новое в knowledge/, но никто не обновляет шлюз. Во-вторых, шлюз — это каталог ссылок, который правила документации прямо запрещают: «CLAUDE.md не должен содержать каталог ссылок на knowledge-файлы — для этого `docs_search`».

Правильная активация — *семантический роутер `docs_search` сам подтягивает 2-4 релевантных файла по симптому запроса*. CLAUDE.md содержит только guard rails (anti-patterns, стек, философские принципы уровня всего проекта), точка. Юзер сформулировал прямо: *«что за хуйня, у нас по правилам документации main»*. То есть `main-*` файлы — артефакт ранней структуры до того как `docs_search` стал основным каналом обнаружения.

В этом проекте: миграция выполнена в текущей сессии. Удалены `architecture.md`, `main-design.md`, `main-feature.md`, `features/`, `ai-research/`. Содержимое разошлось по `knowledge/fact-*` и `methodology/переносимый-дизайн.md` по принципу симптома. CLAUDE.md переписан под Gold Standard v7.0 — `<instructions>` теперь вызывает `docs_search`, не предписывает чтение конкретных файлов.
