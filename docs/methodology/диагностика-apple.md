# Детерминированная диагностика

В диагностических скриптах каждая ошибка должна определяться **специфическим тестом**, а не методом исключения. Юзер: *«А точно ли проблема не с тем, что телефона нет в Xcode? Если способы детерминированно определить, короче, проверь там»*. Если категорию ошибки нельзя подтвердить отдельным сигналом — её нельзя называть в вердикте, какой бы правдоподобной она ни была.

Распространённая ошибка скрипта: автор перечисляет известные причины проблемы (VPN, network, pairing, DDI), и когда не находит ни одной из остальных, **обвиняет первую попавшуюся**. Это и есть метод исключения, и он ведёт к ложным диагнозам. Реальный пример из сессии: скрипт обвинял AmneziaVPN в недоступности iPhone несколько итераций подряд, хотя проблема была в неустановленном `XcodeSystemResources.pkg` после обновления Xcode. Юзер заметил это вопросом *"а точно ли проблема не с тем, что телефона нет"* — после чего удалось добавить специфические тесты (Bonjour reachability, разрешение `iPhone.local`, `devicectl preferredDDI`) и определить корень.

Три правила для диагностических скриптов:

1. **Каждая категория ошибки = отдельный тест.** Не "не найдено A и не найдено B → значит C". Должен быть позитивный тест на C, который даёт `yes/no` независимо от A и B.
2. **Нет специфического теста = категорию не показывай в вердикте.** Лучше вывести `UNKNOWN` или показать сырые данные диагностики, чем угадывать. Юзер сам разберётся.
3. **Категория "информационно" ≠ "виновник".** В сессии VPN-туннель был помечен виновником пять раз подряд, хотя ни один тест не показывал что VPN ломает CoreDevice. Информация о VPN полезна для контекста, но её нельзя класть в `verdict_action` без отдельного позитивного теста, что именно VPN вызывает наблюдаемую ошибку.

В habit-tracker этот принцип воплощён в `deploy.sh` — см. таблицу "Категория ошибки → специфический тест" в `knowledge/fact-wireless-deploy.md`.

## Идемпотентность tooling: restore initial state, don't enforce default state

Скрипт-обёртка, который поднимает побочный процесс ради своей задачи, обязан **вернуть систему в то состояние, в котором её получил**. Если процесса не было — закрыть его на выходе. Если процесс был — не трогать. Юзер сформулировал это прямо: «я хочу что [скрипт] запускал то что нужно и закрывал xcode», и в той же сессии — контраст: «если xcode был уже запущен юзером — не трогать». Это не "default state = Xcode closed", это "snapshot at entry, restore at exit".

Конкретный механизм в `deploy.sh`: snapshot `XCODE_WAS_RUNNING` в начале через `pgrep`, `trap cleanup_xcode EXIT INT TERM` который закрывает Xcode **только если сам поднял** (`XCODE_WAS_RUNNING=false`). AppleScript `quit` асинхронный — обязателен poll-wait до 8с пока процесс реально умрёт, иначе следующий вызов скрипта найдёт Xcode "running" и подумает что юзер его открыл (false-positive). Hard-kill через `pkill` — только fallback после wait timeout, не первичный путь (мягкое завершение даёт Xcode сохранить state).

Антипаттерн который ломает принцип: «всегда закрывай Xcode на выходе для чистоты». Это enforces default state — юзер открыл Xcode для своей работы, скрипт его закрыл, юзер выматерился. Snapshot-at-entry / restore-at-exit сохраняет агентность юзера.

## Непрозрачный баг: research-гипотеза → таргетные логи → наблюдение → итерация

Когда фича не работает и причина не видна из кода (сетевое поведение, поведение платформы, гонки на чужой стороне), петля не «угадал причину → починил», а итеративная: юзер сформулировал её прямо — *«искать в интернете гипотезу, добавлять логи на основе логов и расширять своё знание, и искать дальше»*. Сначала ресёрч даёт гипотезу о механизме, затем под эту гипотезу ставятся **точечные логи именно на сомнительный шаг** (не «логировать всё»), затем наблюдается реальный цикл работы, и его данные сужают следующую гипотезу. Знание расширяется до починки, а не до правдоподобного объяснения.

Ключевое отличие от метода исключения (секция выше): там запрещено называть причину без позитивного теста; здесь позитивный тест **строится итеративно** — лог добавляется не чтобы подтвердить готовый вердикт, а чтобы увидеть какой шаг расходится с гипотезой. Полезно разворачивать ресёрч-гипотезы фан-аутом независимых проверок с адверсариальной верификацией (скептик пытается опровергнуть гипотезу до того как она ляжет в основу починки) — иначе правдоподобная-но-ложная гипотеза переживает один цикл и уводит отладку. В habit-tracker реальный пример — диагностика «фича не будит Mac»: ресёрч дал гипотезу про истечение ARP, живой замер сети её подтвердил, а логирование `viability` для UDP вскрыло невидимый инвариант (сокет застывает на `.ready`), которого в коде не видно. См. `knowledge/fix-пробуждение-mac.md`.

## Фриз требует наблюдателя вне замёрзшего цикла

UI-freeze нельзя диагностировать только логами "рядом" с подозрительным кодом: если main
thread или render pipeline зависает, обычный in-process лог как раз может не выполниться в
момент проблемы. Нужен хотя бы один наблюдатель другого класса. Для main-thread hang это
фоновый watchdog, который пингует main через semaphore и пишет `[hang]` уже после
восстановления; для render/server-композиции это `CADisplayLink`, который видит реальные
gaps между предъявленными кадрами даже когда main формально свободен. Если watchdog молчит,
а display link показывает dropped frames, диагноз "main заблокировался" запрещён.

Вторая половина правила — фазовая атрибуция. Одно число `worst=79ms` говорит только что кадр
был плохой; оно не говорит, что чинить. Навигационные/анимационные окна нужно размечать
фазами (`mount`, `animate`, `swap`, `settle`) и привязывать к `nav#`/label действия. Тогда
"клик по проекту лагает" раскладывается на конкретный слой: mount тяжёлый до движения,
animate тяжёлый во время offset, swap тяжёлый при live-коммите, settle тяжёлый при дорисовке
controls. В habit-tracker это стало обязательным для Terminal навигации; конкретный шрам —
`knowledge/fix-ios-stability.md::Terminal navigation jank`.
