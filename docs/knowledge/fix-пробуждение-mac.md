# Пробуждение Mac с iPhone (Wake-on-LAN) — отклонено

Пробовали кнопку «Разбудить Mac» в Voice Settings (magic packet с iPhone + карточка статуса активен/спит/недоступен). **Фича удалена из приложения** — не определяла спящий Mac надёжно и не будила его после ночи. Юзер выбрал другое решение: не усыплять MacBook (держать на зарядке, не давать уходить в сон), тогда будить вообще не нужно.

Файл сохранён как graveyard: ниже — почему on-device WoL не работает, чтобы не переизобретать фичу в следующий раз. Кода в проекте больше нет (`MacWakeManager.swift` / `MacWakeSection.swift` удалены).

**Главное: надёжно разбудить спавший несколько часов Mac с iPhone (free Apple ID, без доп. железа) НЕЛЬЗЯ.** Ниже — три независимые причины.

## Стена: on-device WoL не будит спящий Mac надёжно

Проверено ресёрчем (адверсариальная верификация) + живым замером ARP на сети. Три независимые причины, каждая фатальна:

1. **Unicast умирает на истёкшем ARP.** Magic packet шлётся UDP-unicast'ом на IP Mac (не broadcast — см. ниже почему). Роутер строит L2-кадр через ARP-резолв IP→MAC. Спящий хост перестаёт отвечать на ARP, через минуты запись в ARP-кэше роутера становится `incomplete`, и unicast-кадр **тихо дропается, не дойдя до сетевой карты**. Работает только первые ~2-5 минут после засыпания, пока запись ещё в кэше. **Тест соврёт**: усыпил → нажал через минуту → проснулся; после ночи — тишина. Живой замер показал ровно это: `arp -a` дал `192.168.0.102 → (incomplete)` для уехавшего хоста.

2. **Broadcast закрыт entitlement'ом, недоступным free Apple ID.** Классический фикс ARP-проблемы — слать на subnet-broadcast (`192.168.0.255`) или `255.255.255.255`, broadcast переживает истечение ARP. Но iOS гейтит broadcast по **broadcast-биту, не по литералу адреса** — и `192.168.0.255`, и `255.255.255.255` одинаково требуют `com.apple.developer.networking.multicast`. Это managed-entitlement, Apple **не подписывает его free Apple ID** (проект деплоит free ID, см. `fact-wireless-deploy.md`). Неэнтайтленный broadcast ядро дропает как **errno 65 «No route to host»** — легко принять за проблему маршрутизации/Wi-Fi.

3. **Sleep proxy в сети нет.** Bonjour Sleep Proxy (Apple TV / HomePod / always-on Mac) держал бы ARP-запись спящего за себя через gratuitous ARP — тогда unicast дошёл бы. Замер сети (`dns-sd -B _sleep-proxy._udp`) показал: прокси нет, в LAN только роутер TP-Link, этот Mac и один телефон.

**Единственный надёжный путь** — always-on устройство на той же LAN (второй Mac / Pi / NAS), куда iPhone стучит обычным unicast'ом, и оно шлёт настоящий L2-broadcast локально. На iOS из песочницы L2-broadcast не отправить (нет доступа к сырым кадрам, MAC назначения берётся из ARP и не контролируется приложением).

### Что пробовали и почему не сработало
- **Unicast на IP** — доставляется только пока ARP-запись жива (~5 мин), потом дроп. Best-effort, не overnight.
- **Subnet-directed broadcast `.255`** — НЕ лазейка, тот же entitlement, errno 65.
- **L2-broadcast MAC `FF:FF:FF:FF:FF:FF` напрямую** — невозможно, iOS не даёт raw-frame инъекцию без root.

## Детект сна

**UDP NWConnection застывает на `.ready` и не уходит в `.failed` при мёртвом маршруте** — поэтому `stateUpdateHandler` СЛЕП к «пакет не ушёл». Единственный живой сигнал — `viabilityUpdateHandler`: send-OK с `viable=false` = пакет ушёл в мёртвый маршрут (доставки нет), отличается от реального ухода. Поэтому в `wake()` viability логируется рядом с send.

**Проба статуса — TCP-connect на порт 22 (SSH), и тут контринтуитив** (если будешь делать детект Mac снова):
- `ECONNREFUSED` (errno 61) = Mac **АКТИВЕН**, просто Remote Login выключен. Наивный мап `.failed → .asleep` — баг: включённый Mac без SSH покажется спящим. Правильно: `ECONNREFUSED → awake`, только `ETIMEDOUT/EHOSTUNREACH/EHOSTDOWN/ENETUNREACH → asleep`.
- TCP-проба **не даёт ложного «активен»**: спящий хост физически не завершит handshake, успешный connect = Mac реально проснулся в этот момент. (Изначальное опасение «dark wake даст ложный awake» — неверно.)
- Зато ICMP ping / mDNS browse / ARP presence **дают ложный «активен»** при наличии sleep proxy — он отвечает за спящего. Поэтому проба именно по TCP-сервису, не пинг.
- Нюанс: для порта 22 sleep proxy будит Mac на любой пакет, не только SYN — то есть проба :22 это активное действие пробуждения, не пассивное чтение. На нашей LAN без прокси неважно.

## Настройка на Mac (платформенные факты)

- `sudo pmset -c womp 1` — Wake on Magic Packet, **только на зарядке**. На батарее `womp 0` оставлен намеренно: Wi-Fi-чип в глубоком сне на батарее всё равно не ловит пакеты, только разряд.
- **`standbydelay`, `destroyfvkeyonstandby`, `proximitywake` — Intel-only.** На Apple Silicon `pmset` принимает их (exit 0), но ключи **не появляются в `pmset -g custom`** и не действуют — модель сна M-серии управляется иначе (Secure Enclave). Не закладываться на них.
- **Private Wi-Fi Address для домашнего SSID должен быть Off** — иначе MAC рандомизируется per-network и зашитый в приложение MAC не совпадёт. Реальный MAC берётся из `ifconfig en0` (Wi-Fi → ⓘ сети → Адрес Wi-Fi с Off).

## DHCP-ротация ломает зашитый IP

Симптом: вчера будил, сегодня «недоступен». Причина: IP зашит хардкодом, DHCP-роутер выдал Mac новый адрес за ночь (наблюдали `.102 → .103`). Поэтому IP в настройках сделан редактируемым. Надёжные фиксы (не внедрены):
- **Резолв `MacBook-Air-Fedor.local`** вместо IP — но на сети без sleep proxy спящий Mac не отвечает на mDNS, имя резолвится только пока Mac активен. Годится для пробы статуса, бесполезно как цель пробуждения.
- **Address Reservation на роутере** (TP-Link → DHCP Server → Address Reservation, привязать MAC к IP) — стабильная цель для WoL, не зависит от mDNS во сне.

## Tahoe FileVault unlock over SSH (исследовано, не внедрено)

macOS 26 Tahoe на Apple Silicon умеет принимать SSH в pre-boot окружении и разблокировать FileVault вводом пароля удалённо (`apple_ssh_and_filevault`). Раньше было невозможно — на FV-экране сети не было. Подводные камни: pre-login SSH на Wi-Fi нестабилен, надёжно только по Ethernet (до разблокировки keychain Wi-Fi не поднят). Для overnight реалистичнее держать `destroyfvkeyonstandby 0` (wake идёт в обычный login-экран с сетью, а не в FV-экран) — но на Apple Silicon этот ключ игнорируется (см. выше), так что поведение FV-экрана не настраивается этим путём.
